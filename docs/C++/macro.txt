预处理宏 vs 普通变量
================================================================================

一、处理阶段
================================================================================

C/C++ 源文件从代码到可执行文件经过四个阶段：

  源文件(.cu/.cpp)
      │
      ▼
  【预处理阶段】← 预处理宏在此处理
      预处理器（cpp）执行所有 # 指令：
        #include  → 将头文件内容原样粘贴进来
        #define   → 注册文本替换规则
        #ifdef    → 按宏是否存在决定保留哪段代码
      输出：展开后的纯 C++ 文本，不含任何 # 指令
      │
      ▼
  【编译阶段】← 普通变量在此处理
      编译器（g++/nvcc）将 C++ 文本翻译为机器指令（.o 目标文件）
      类型检查、作用域分析、模板实例化均在此阶段
      │
      ▼
  【汇编阶段】
      汇编器将汇编文本转换为二进制机器码
      │
      ▼
  【链接阶段】
      链接器合并多个 .o 文件，解析符号地址，生成 .so 或可执行文件


二、本质区别
================================================================================

┌─────────────────┬────────────────────────────┬────────────────────────────┐
│                 │ 预处理宏 #define            │ 普通变量                   │
├─────────────────┼────────────────────────────┼────────────────────────────┤
│ 处理阶段        │ 预处理阶段（编译前）        │ 编译阶段                   │
│ 有类型吗        │ 没有                        │ 有（int / float / ...）    │
│ 有内存地址吗    │ 没有                        │ 有                         │
│ 有作用域吗      │ 没有，定义后全文件可见      │ 有（函数 / 块作用域）      │
│ 能取地址吗      │ 不能                        │ 能（&var）                 │
│ 能调试器查看吗  │ 不能（已被替换，不存在了）  │ 能（gdb p var）            │
│ 本质是什么      │ 预处理器的文本替换规则      │ 编译器管理的具名存储单元   │
└─────────────────┴────────────────────────────┴────────────────────────────┘


三、#define 的工作方式
================================================================================

#define SOFTMAX_VARIANT 8

  预处理器扫描到这一行时：
    1. 在符号表中登记：SOFTMAX_VARIANT → 文本 "8"
    2. 继续扫描后续代码，遇到 SOFTMAX_VARIANT 就原样替换为 8

  编译器看到的是替换后的结果，根本不知道 SOFTMAX_VARIANT 这个名字存在过：

    源码：                         预处理后编译器看到的：
    ─────────────────────────      ──────────────────────────────────
    #define SOFTMAX_VARIANT 8      （此行消失）
    softmax_kernel<SOFTMAX_VARIANT> → softmax_kernel<8>
    int x = SOFTMAX_VARIANT + 1;  → int x = 8 + 1;

  关键点：
    - 替换是纯文本操作，不做类型检查
    - 预处理后宏名消失，调试时无法查询 SOFTMAX_VARIANT 的值
    - 替换发生在词法层面，不理解 C++ 语义


四、#define vs const int
================================================================================

  #define BLOCK_DIM_Y 1024          // 宏：无类型，文本替换，无地址
  const int BLOCK_DIM_Y = 1024;     // 常量变量：有类型 int，有地址，有作用域

  两者定义"常量"的效果相似，但：
    - const int 有类型，编译器能做类型检查和隐式转换检测
    - const int 有地址，可取地址（&BLOCK_DIM_Y）
    - const int 有作用域，可放在命名空间 / 函数内部
    - #define 作用域从定义处到文件末尾（或 #undef），跨越所有函数
    - #define 可被编译器的 -D 标志从外部覆盖，const int 不能

  现代 C++ 推荐用 constexpr 替代宏常量：
    constexpr int BLOCK_DIM_Y = 1024;   // 编译期常量，有类型，有作用域


五、#ifndef 保护 + 外部覆盖模式
================================================================================

  本项目 kernels.cu 中的典型写法：

    #ifndef SOFTMAX_VARIANT
    #define SOFTMAX_VARIANT 8     // 若没有外部传入，使用此默认值
    #endif

    #ifndef BLOCK_DIM_Y
    #define BLOCK_DIM_Y 1024
    #endif

  逻辑：
    #ifndef SOFTMAX_VARIANT  → "如果 SOFTMAX_VARIANT 还没有被定义"
    #define SOFTMAX_VARIANT 8 → "则现在定义它为 8"
    #endif                   → "条件结束"

  若编译器已收到 -DSOFTMAX_VARIANT=6：
    等价于在文件第一行插入  #define SOFTMAX_VARIANT 6
    后续 #ifndef SOFTMAX_VARIANT 条件为假（已定义），跳过内部默认值
    最终使用 6

  若编译器未收到任何 -DSOFTMAX_VARIANT：
    #ifndef 条件为真（未定义），执行 #define SOFTMAX_VARIANT 8
    最终使用默认值 8

  优先级总结：
    编译器 -D（外部传入）  >  #ifndef 保护的默认值


六、两个 -D 标志：cmake -D 和 编译器 -D
================================================================================

  相同写法，不同系统，含义完全不同：

  【cmake -DSOFTMAX_VARIANT=6】
    - 属于 CMake 变量系统
    - 作用：在 CMake 配置阶段定义一个名为 SOFTMAX_VARIANT 的 CMake 变量，值为 "6"
    - 存储：写入 CMakeCache.txt，跨 Reload 持久保存
    - 此时源代码的 #ifndef 还看不到它，它只是 CMake 的一个字符串变量

  【nvcc/g++ -DSOFTMAX_VARIANT=6】
    - 属于 C 预处理器系统
    - 作用：告诉预处理器"在处理源文件前先执行 #define SOFTMAX_VARIANT 6"
    - 等价于在源文件第一行插入 #define SOFTMAX_VARIANT 6
    - 此后 #ifndef SOFTMAX_VARIANT 条件为假，跳过默认值

七、本项目中的宏列表
================================================================================

  通过 target_compile_definitions 注入，均可由 cmake -D 从外部覆盖：

  ┌──────────────────────┬──────────────┬──────────────────────────────────────┐
  │ 宏名                 │ 默认值       │ 用途                                 │
  ├──────────────────────┼──────────────┼──────────────────────────────────────┤
  │ SOFTMAX_VARIANT      │ 8            │ 选择 kernel 实现（1~9）              │
  │ BLOCK_DIM_Y          │ 1024         │ 每 block Y 维线程数，必须是 2 的幂   │
  │ UNROLL_FACTOR        │ 8            │ 循环展开因子（1/2/4/8）              │
  │ WIDTH                │ 0            │ 行宽静态提示，0 = 运行时参数决定     │
  │ TORCH_EXTENSION_NAME │ softmax_cuda │ Python 模块名，PYBIND11_MODULE 使用  │
  │ _GLIBCXX_USE_CXX11_ABI│ 1           │ GCC ABI 版本，由 TORCH_CXX_FLAGS 注入│
  └──────────────────────┴──────────────┴──────────────────────────────────────┘

  使用示例（选择 variant=6，block_y=512）：
    cmake -DSOFTMAX_VARIANT=6 -DBLOCK_DIM_Y=512 ..


八、为什么 AT_DISPATCH_FLOATING_TYPES 必须是宏而非函数
================================================================================

  AT_DISPATCH_FLOATING_TYPES 展开后的结构（简化）：

    case at::ScalarType::Float: {
        using scalar_t = float;        // ← 宏注入的类型别名
        softmax_kernel<scalar_t><<<...>>>(x.data_ptr<scalar_t>(), ...);  // ← lambda 体原样内联
    }

  核心问题：using scalar_t = float 是编译期的类型别名，不是运行时值：
    - 没有内存地址，无法被 lambda [&] 捕获
    - 只存在于当前词法作用域（{}块）内
    - lambda 体和 using 声明必须在同一 {} 块内，scalar_t 才可见

  若改为函数：

    void dispatch(std::function<void()> fn) {
        using scalar_t = float;    // 定义在函数体内
        fn();                      // fn 是另一个函数，scalar_t 对 fn 不可见
    }
    dispatch([&]{ softmax_kernel<scalar_t><<<...>>>(...); });
    //                            ^ 编译错误：scalar_t 未定义

  只有宏（预处理器文本替换）才能将 using scalar_t = float 和调用代码
  放入同一个词法作用域，这是 AT_DISPATCH 必须是宏的根本原因。

  详细说明见：docs/AT_DISPATCH_FLOATING_TYPES.txt 【3.2 scalar_t 的可见性】
